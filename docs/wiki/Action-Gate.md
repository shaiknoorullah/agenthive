# Action Gate

The action gate is the mechanism that lets agents (Claude Code, Codex, etc.) ask "may I run this?" and have *any* device — phone, laptop, dev server — answer first. It's a file-queue + atomic-create + blocking-poll pattern. The cleverness is all in one Unix flag.

## The pattern

```mermaid
sequenceDiagram
    autonumber
    participant Hook as agenthive hook
    participant Queue as Local file queue
    participant Daemon as Local daemon
    participant Surfaces as Surface 1..N

    Hook->>Queue: write &lt;id&gt;.pending (atomic temp + rename)
    Hook->>Daemon: action_request via Unix socket
    Daemon->>Surfaces: fan out via libp2p streams + local dispatch
    Hook->>Queue: poll for &lt;id&gt;.response
    Surfaces->>Queue: O_CREAT|O_EXCL &lt;id&gt;.response
    Note over Surfaces,Queue: First wins. Others get EEXIST.
    Queue-->>Hook: response observed
    Hook->>Queue: delete &lt;id&gt;.pending and &lt;id&gt;.response
    Hook-->>Agent: hook output JSON
```

## File queue layout

```
<config-dir>/actions/
├── a1b2c3d4.pending    # The pending request
└── a1b2c3d4.response   # The decision, when one exists
```

`<id>` is a 32-character hex random ID generated by `crypto/rand` inside the hook subcommand. Probability of collision is negligible.

### Pending file format

```json
{
  "action_id":   "a1b2c3d4...",
  "session_id":  "...",
  "tool_use_id": "...",
  "tool_name":   "Bash",
  "tool_input":  "rm -rf /tmp/build",
  "project":     "...",
  "cwd":         "...",
  "timestamp":   "2026-06-04T01:00:00Z",
  "expires_at":  "2026-06-04T01:05:00Z"
}
```

### Response file format

```json
{
  "action_id":  "a1b2c3d4...",
  "decision":   "allow",
  "decided_by": "12D3KooWPhone..."
}
```

## The atomicity primitive

A response writer (any surface — TUI, phone, manual `agenthive respond`) calls:

```go
f, err := os.OpenFile(path, os.O_WRONLY|os.O_CREATE|os.O_EXCL, 0600)
```

`O_CREAT|O_EXCL` is a kernel-enforced atomic operation: either the file is created and `f` is a valid handle, or the file existed already and you get `syscall.EEXIST`. No partial state, no TOCTOU window. The first writer wins; everybody else gets a clean failure they can detect.

This is the **only** reason agenthive can fan out the same request to multiple devices and not race. Without `O_CREAT|O_EXCL`, two devices clicking Allow at the same millisecond would both write a response and the gate would observe whichever one happened to overwrite last. With it, the second writer learns it lost the race and dismisses its own prompt.

## TTL and destructive actions

Every pending action carries an `expires_at`. Two TTLs:

- **Normal**: 300s (5 minutes)
- **Destructive**: 30s

The destructive flag is set by `internal/hooks/security.go::Classify`, which case-insensitively substring-matches the combined `tool_name + tool_input` against a curated list:

```
rm -rf, rm -fr, git push --force, git push -f, git reset --hard,
git clean -f, DROP TABLE, DELETE FROM, TRUNCATE, reboot, shutdown,
chmod 777, chown -R, dd if=, mkfs, > /dev/, :>!, rm -- *
```

Substring matching is deliberate. `rm -rf .DS_Store` still trips the path because an extra prompt is cheaper than a missed `rm -rf`.

When the gate's poll loop sees `expires_at` pass without a response, it returns `context.DeadlineExceeded` to the hook subcommand. The hook then exits 0 with **no output** so Claude Code falls back to its built-in permission prompt — never fail-closed.

## Polling cadence

The gate polls for `<id>.response` at a 50ms base interval, doubling up to a 500ms cap. Typical response from a phone arrives in 1–3 seconds; from the TUI in well under 100ms.

The poll is on the file system, not on the network. The network leg is "publish action_request → wait for any surface to write response file", which is just file-system observation. No need for the gate to read from inbound libp2p streams.

## Fail-open guarantee

Every error in the gate path returns the hook to fail-open:

| What fails | Hook behavior | Claude behavior |
|---|---|---|
| Daemon socket unreachable | Print nothing, exit 0 | Built-in permission prompt |
| Gate write to `<id>.pending` fails | Print nothing, exit 0 | Built-in permission prompt |
| Gate poll exceeds TTL | Print nothing, exit 0 | Built-in permission prompt |
| Response file is malformed JSON | Print nothing, exit 0 | Built-in permission prompt |

This is non-negotiable. A broken agenthive **must not** break Claude's existing permission UX.

## Manual override

Use `agenthive respond <action-id> allow|deny` from any terminal to write a response from outside any surface. Useful for debugging, scripted approvals (e.g., a CI bot), or end-to-end tests.

## Cleanup

After the gate observes a response, it deletes both `<id>.pending` and `<id>.response`. Files that age past `expires_at + 5 minutes` are also swept on the next gate run.

In v0.1.0 the sweep is on demand. A periodic janitor goroutine ships in v0.2.0.
